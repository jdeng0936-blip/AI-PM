# 徽远成 AI-PM（智能项目经理）— 白皮书·总体设计

> **版本**：v1.1 | **日期**：2026-03-10 | **来源**：基于 Gemini 共享文档提炼 + 实际开发反馈更新

## 一、项目愿景与背景

### 1.1 痛点分析

传统项目管理依赖 12 列 Excel 日报，存在以下核心问题：

| 痛点 | 表现 |
|------|------|
| 录入反人类 | 员工面对 12 个格子抓耳挠腮，日报敷衍、大量留白 |
| 检查靠人工 | 主管需逐条审阅，耗时且易遗漏逻辑漏洞 |
| 监督无连续性 | 跨日任务追踪断裂，卡点无人跟进 |
| 评价主观化 | 缺乏量化标准，考核失去公信力 |
| 信息碎片化 | CEO/CTO 需翻阅大量微信消息才能了解全局 |

### 1.2 系统定位

**"徽远成 AI-PM"** 是一套以 **大模型（Gemini）** 为核心引擎的智能项目管理系统，目标是：

> **让日报录入从 5 分钟缩短到 10 秒，让管理从"人盯人"升级为"AI 驱动的自动化 PDCA 闭环"。**

### 1.3 团队优势

- 团队已深度使用大模型（张毅做 NewApi/Liberchat 接入、郭震用 Antigravity 写代码、陈家云用智能体优化仓库）
- 具备 "Eat Your Own Dog Food" 的内部验证能力
- 技术栈前沿、软硬结合，可快速自研落地

## 二、核心功能模块设计

系统围绕 **PDCA（Plan-Do-Check-Act）** 管理闭环，设计了 **五大核心模块**：

```
┌─────────────────────────────────────────────────────┐
│                 AI-PM 系统架构总览                     │
├─────────────────────────────────────────────────────┤
│                                                     │
│   ① 录入(Input)  ──→  ② 检查(Check)  ──→            │
│        ↑                                            │
│        │           ③ 监督(Supervise)                  │
│        │                │                            │
│        └────────────────┘                            │
│                                                     │
│   ④ 评分(Score)    ⑤ 管理(Manage)                     │
│      ↓                  ↓                            │
│   游戏化激励          战情看板 + 自动周报               │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 模块 ① — 双模式录入 (Plan & Review)

**核心理念**：从"填 12 个格子"变成"选模式 → 填字段 → 提交"

| 要素 | 设计方案 |
|------|---------|
| **交互入口** | Web管理台 / Liberchat / 钉钉·企微机器人 |
| **输入方式** | 📋结构化表单 / ✏️自由文本 / 🎤语音输入 |
| **双模式** | ☀️晨规划（12:00前）/ 🌙晚复核（12:00后）自动切换 |
| **AI 解析引擎** | Mock AI 引擎（开发） / Gemini 2.5 Flash（生产） |
| **效率提升** | 5 分钟 → 30 秒（结构化表单引导） |

**☀️ 晨规划字段**：

| 字段 | 必填? | 评分影响 |
|------|-------|--------|
| 📌 今日计划 | **必填** | 描述长度 +5/+10 |
| 🎯 目标描述 | **必填** | — |
| 📊 进度预期 | **必填** | 进度值 +5/+10 |
| ✅ 验收标准 | 选填(加分) | +8分 |
| 📦 预期交付 | 选填(加分) | +5分 |
| 🤝 需要协助 | 选填 | — |

**🌙 晚复核字段**（7个）：完成任务/实际进度/代码提交/验收结果/验收人/遗留卡点/所需支持

### 模块 ② — 智能质检闭环 (Check)

**核心理念**：累加评分制 + 驳回修改闭环 — 未通过不入库

**评分规则（累加制）**：

| 评分项 | 分值 | 说明 |
|--------|------|------|
| 基础分 | 50 | 提交即得 |
| 详细描述 (>100字) | +5 | 核心 |
| 更详细 (>200字) | +5 | 建议 |
| 包含量化数字 | +3 | 如"完成80%" |
| 进度 ≥80% | +10 | 或 ≥50% 得+5 |
| 验收标准 | +8 | 加分项(非必填) |
| 交付物/成果 | +5 | 加分项 |
| Git 版本号 | +5 | 仅代码项目，非必填 |
| 卡点详细说明 | +5 | 加分项 |

**通过阈值**：**≥60分** → ✅ 入库 · **<60分** → 🚫 驳回

**驳回闭环**：
- 驳回后显示：❌ 驳回原因 + 📝 修改建议（区分"核心问题"和"加分建议"）
- 用户修改后一键重新提交（内容保留在输入框中）
- Git版本号标注"非研发岗位可忽略"，避免误导
- 未通过的报告**不写入 daily_reports**，仅记录 token 用量

---

### 模块 ③ — 跨日监督 (Supervise)

**核心理念**：AI 充当"不知疲倦的敏捷教练"，具备上下文理解能力

| 能力 | 描述 |
|------|------|
| **跨日连贯性监督** | 自动追踪前日卡点，标记为 `[待解决]`，次日重点监督推进情况 |
| **上下游依赖预警** | 识别任务间依赖关系，当上游阻塞时自动提醒下游调整节奏 |

**场景示例**：

- **跨日追踪**：陈家云 25 日提到"原材料仓缺实操"，AI 自动在 26 日监督该问题是否推进
- **依赖预警**：林跃文（采购 MCU/LoRa 模块）的进度直接影响 206 样机贴片与测试。若采购延迟，AI 自动给下游发预警

### 模块 ④ — 游戏化评分 (Score)

**核心理念**：多维度量化打分 + "反卷"提醒

**评分维度**：

| 维度 | 权重 | 说明 |
|------|------|------|
| 日报含金量 | 高 | 是否有实质产出描述 |
| 逻辑严密性 | 高 | 进度、卡点、方案是否自洽 |
| 规范完整度 | 中 | 必填字段是否齐全 |
| 协作贡献度 | 中 | 是否主动协调跨部门问题 |

**评分输出示例**：

```json
{
  "员工": "陈家云",
  "日期": "2月25日",
  "评分": 85,
  "等级": "良好",
  "AI点评": "工作态度积极，主动协调原材料问题，但在解决问题时效性上需在26日进一步明确节点",
  "推荐": "当日表现优秀员工",
  "管理警告": "仓库缺实操环境，可能导致后续流程阻塞，请主管郭震协助解决"
}
```

### 模块 ⑤ — 战情管理 (Manage)

**核心理念**：老板的"战情看板" — 每日 9:00 自动推送

| 功能 | 描述 |
|------|------|
| **项目健康度概览** | 🔴 红（卡点严重/延期）🟡 黄（依赖项异常）🟢 绿（正常推进）三色仪表盘 |

| **自动化周报** | 从日报中自动提炼关键里程碑 |

### 3.3 核心 Prompt 工程

系统的 AI 行为由精心设计的 System Prompt 驱动：

```python
system_prompt = """
你现在是徽远成科技的"AI 项目经理"。请根据以下员工提交的单日工作记录JSON，
进行严格的检查、指出问题，并给出一个0-100分的综合评分。

【检查规则】
1. 逻辑校验：如果"完成进度"不足100%，但"核心卡点"为"无"或空，
   判定为逻辑不合格(pass_check: false)，必须扣分。

【返回格式要求 (必须是严格的JSON)】
{
  "pass_check": true/false,
  "reject_reason": "如果不通过，请用温和的语气指出需要员工补充什么字段",
  "score": 85,
  "ai_comment": "作为主管的一段点评（50字以内，包含肯定与建议）",
  "management_alert": "如果有进度严重滞后或需要主管协调资源，
                       请在此输出警告提取，否则为空"
}
"""
```

## 五、长期研发项目跟踪（Project Lifecycle）

> 前面五大模块解决的是**日报级**的短周期管理。对于硬件研发、嵌入式开发等**跨月甚至跨季度**的长期项目，需要额外的迭代跟踪机制。

### 5.1 设计理念

日报是"每日快照"，但研发项目的生命周期远超一天。系统需要一个**纵向追踪层**，将零散的日报自动聚合为项目全景：

```
日报(Day) → Sprint(周/双周) → 里程碑(Milestone) → 项目全生命周期(Project)
```

### 5.2 核心能力

#### ① Sprint / 迭代管理

| 能力 | 描述 |
|------|------|
| **自动迭代划分** | 按周/双周自动生成 Sprint，支持自定义迭代周期 |
| **日报自动归集** | 员工日报自动按 Sprint + 所属任务归集，无需二次操作 |
| **燃尽图自动生成** | 根据每日进度数据自动绘制 Sprint 燃尽图 |
| **迭代回顾 AI 总结** | Sprint 结束时，AI 自动生成回顾报告（完成率、延期原因、改进建议） |

#### ② 里程碑 & WBS 分解

| 能力 | 描述 |
|------|------|
| **多级任务分解** | 项目 → 里程碑 → Epic → 任务（支持 WBS 层级拆解） |
| **关键路径识别** | AI 自动识别项目关键路径，标注哪些任务延期会影响整体交付 |
| **依赖链可视化** | 任务间的前置/后置依赖关系图谱，自动检测环形依赖 |

#### ③ AI 趋势分析 & 风险预测

| 能力 | 描述 |
|------|------|
| **进度趋势预测** | 根据历史 Sprint 效率，AI 预测项目整体交付时间 |
| **风险自动识别** | 连续多日进度停滞、卡点重复出现 → AI 自动升级为项目风险 |
| **资源负载均衡** | 分析各成员任务分布，识别过载或闲置，建议任务调配 |
| **技术债追踪** | 标记"临时方案"、"待重构"等技术债，定期提醒处理 |

#### ④ 多项目并行看板

| 能力 | 描述 |
|------|------|
| **项目组合视图** | 多个研发项目并行状态总览（甘特图 + 健康度） |
| **跨项目资源冲突** | 同一成员参与多个项目时，AI 检测时间冲突并预警 |
| **项目对比分析** | 对比不同项目的效率指标（Sprint 完成率、延期频率等） |

### 5.3 数据模型扩展

```
projects (项目)
  ├── milestones (里程碑)
  │     ├── epics (史诗/功能模块)
  │     │     ├── tasks (任务) ← 日报自动关联
  │     │     │     ├── daily_reports (日报记录)
  │     │     │     └── blockers (卡点记录)
  │     │     └── tech_debts (技术债)
  │     └── dependencies (依赖关系)
  ├── sprints (迭代)
  │     ├── sprint_tasks (迭代-任务关联)
  │     └── sprint_reviews (迭代回顾 - AI 生成)
  └── risk_registry (风险登记册 - AI 维护)
```

### 5.4 AI 自动化场景

| 场景 | 触发条件 | AI 动作 | 使用模型 |
|------|---------|--------|---------|
| 迭代回顾 | Sprint 结束 | 生成回顾报告 + 改进建议 | `gemini-3-flash-preview` |
| 风险升级 | 卡点连续 3 天未解决 | 升级为项目风险，通知主管 | `gemini-2.5-flash` |
| 交付预测 | 每周一早 9:00 | 基于历史效率预测交付日期 | `gemini-2.5-pro` |
| 资源告警 | 成员日任务 > 3 项 | 预警过载，建议调配 | `gemini-2.5-flash` |
| 技术债提醒 | 累积 > 5 项未处理 | 建议安排重构 Sprint | `gemini-3-flash-preview` |
| 关键路径更新 | 任何任务延期 | 重新计算关键路径并预警 | `gemini-2.5-flash` |

## 六、支撑体系设计

### 6.1 登录认证与安全

| 要素 | 方案 |
|------|------|
| **认证方式** | 账号密码 + JWT Token 鉴权 |
| **角色区分** | 管理员（总经理）和普通员工两种角色 |
| **会话管理** | Token 有效期 24 小时，支持记住登录 |
| **密码安全** | bcrypt 加密存储，首次登录强制改密 |
| **操作审计** | 记录登录时间、IP、关键操作日志 |

### 6.2 催报机制

未提交日报是系统最大的失效风险，需要自动催报：

| 触发时间 | 动作 | 使用模型 |
|---------|------|---------|
| 每日 17:30 | 友好提醒："今天的日报还没交哦，别忘了～" | — (系统消息) |
| 每日 20:00 | 二次催促："日报截止时间快到了，请尽快提交" | — (系统消息) |
| 每日 22:00（截止） | 标记为"未提交"，通知总经理 | `gemini-2.5-flash` (生成缺勤摘要) |
| 连续 2 天未交 | AI 自动升级预警给总经理 | `gemini-2.5-flash` |

### 6.3 通知推送渠道

| 通知类型 | 推送渠道 | 接收人 |
|---------|---------|-------|
| 日报催提醒 | 企微/钉钉机器人 | 未提交员工 |
| 质检打回 | 企微/钉钉私聊 | 被打回员工 |
| 评分结果 | 系统内通知 | 员工本人 |
| 战情日报 | 企微/钉钉 + 邮件 | 总经理 |
| 风险预警 | 企微/钉钉紧急通知 | 总经理 |
| Sprint 回顾 | 邮件 + 系统内 | 总经理 + 相关人员 |

### 6.4 岗位日报模板差异

不同岗位的工作内容差异巨大，需要按岗位定制日报模板：

| 岗位 | 特有字段 | 示例 |
|------|---------|------|
| **研发人员** | 代码版本号、技术方案、测试结果 | "v2.1 分支已通过单元测试" |
| **采购经理** | 供应商、物料名称、采购金额、到货日期 | "MCU 芯片×500 已下单，预计3天到" |
| **生产经理** | 产线编号、产量、良品率、设备状态 | "A线今日产出200台，良品率98%" |
| **仓管** | 入库/出库明细、库存预警 | "入库 MCU×500，当前库存告警：电阻不足" |
| **财务部长** | 收支明细、发票状态、预算执行率 | "本月预算执行率 72%，应收账款3笔" |
| **商务/销售** | 客户名称、跟进阶段、预计签约金额 | "XX科技 二次报价已发，预计本月签约" |
| **技术部长** | 技术评审、架构决策、技术风险 | "完成 LoRa 模块选型评审" |

AI 解析时会根据提交人的岗位自动匹配对应模板字段。

### 6.5 附件与多媒体支持

| 类型 | 使用场景 | 存储方式 |
|------|---------|---------|
| 📷 图片 | 仓管拍照验货、生产现场拍照 | 阿里云 OSS（数据库只存 URL） |
| 📄 文档 | 采购合同、技术方案 PDF | 阿里云 OSS |
| 🎤 语音 | 语音录入原始文件 | 阿里云 OSS + 讯飞 ASR 转文本 |

> 遵循架构红线 Rule 3：文件二进制严禁入库，一律存 OSS。

### 6.6 总经理 AI 对话查询

总经理可以用自然语言向 AI 提问，系统通过 RAG + SQL 实时检索回答：

| 问题示例 | AI 处理方式 | 使用模型 |
|---------|-----------|---------|
| "本周谁延期最多？" | SQL 聚合查询 → LLM 汇总 | `gemini-3-flash-preview` |
| "采购部进度怎么样？" | 按部门筛选 → LLM 分析 | `gemini-3-flash-preview` |

## 七、战略增强层

> 前六章解决了"怎么把事情做对（执行）"，本章解决"做的事情对不对（战略）"。

### 7.1 OKR 战略对齐

**痛点**：研发人员只盯自己的模块，不知道日常工作对公司商业目标有何贡献。

**方案**：在系统顶层引入 OKR（目标与关键结果）层，将每日任务与公司战略绑定。

| 层级 | 示例 |
|------|------|
| **Objective (公司)** | 206 样机具备行业参展能力 |
| **Key Result** | 大模型推理延迟降低至 500ms 内 |
| **Sprint 任务** | 优化模型推理管线，实现 batch 推理 |
| **日报** | "今日完成 batch 推理优化，延迟从 800ms 降至 450ms" |

**系统能力**：
- 任务创建时可绑定 KR，日报自动关联 OKR 进度
- 总经理看板增加 **OKR 进度仪表盘**：每个 KR 的完成百分比
- AI 周报自动汇总 OKR 推进情况，识别落后的 KR
- 员工每日看到自己的工作在推动哪个 Objective，**提升内驱力**

**数据模型**：

```
objectives (公司级目标)
  ├── key_results (关键结果 - 可量化)
  │     ├── kr_task_links (KR ↔ 任务绑定)
  │     └── kr_progress_logs (KR 进度记录 - AI 从日报自动更新)
  └── okr_cycles (OKR 周期: 季度/半年)
```

### 7.2 资源负载水位与瓶颈预判 (Capacity Planning)

**痛点**：当前系统是"出了卡点才报警（滞后指标）"。多项目并行时，关键人员被过度调度，导致隐性延期风险。

**方案**：引入 **故事点（Story Points）负载池**，实现前瞻性预判。

| 指标 | 计算方式 | 预警规则 |
|------|---------|---------|
| **个人负载水位** | 当前 Sprint 已分配故事点 / 正常负荷 | > 150% → 🔴 过载预警 |
| **项目资源健康度** | 关键路径上人员的平均水位 | > 120% → 🟡 风险预警 |
| **跨项目冲突** | 同一人在多个项目的总故事点 | > 正常负荷 → ⚠️ 冲突提示 |
| **Sprint 可行性** | Sprint 目标总点数 vs 团队总产能 | > 100% → 规划期即预警 |

**AI 介入时机**：

| 触发 | AI 动作 | 使用模型 |
|------|--------|---------|
| Sprint 规划时 | 分析负载分布，建议人员调配 | `gemini-3-flash-preview` |
| 成员被分配 >150% 负荷 | 发出黄色预警，推荐替代方案 | `gemini-2.5-flash` |
| 检测到跨项目冲突 | 生成冲突报告给总经理 | `gemini-2.5-flash` |
| 每周一 | 全员水位概览 + 瓶颈预测 | `gemini-2.5-pro` |

**水位可视化**：

```
张毅:  ████████████████████░░░░  80% (正常)
郭震:  ████████████████████████████████  160% (🔴 过载!)
陈家云: ██████████░░░░░░░░░░░░░░  40% (空闲)
林跃文: ████████████████████████░  100% (满载)
```

### 7.3 AI 驱动自动化复盘与知识资产 (Automated Retrospectives)

**痛点**：Sprint 跑完就完了，踩过的坑在下一个项目里继续踩。传统复盘会又臭又长，大家都不爱开。

**方案**：AI 自动分析历史数据，生成结构化复盘报告，将个人经验转化为公司永久数字资产。

**AI 自动复盘流程**：

```
阶段结束（Sprint / 里程碑 / 项目）
    ↓
AI 自动抽取该阶段全部数据：
  - 所有日报卡点记录
  - 任务延期时间线
  - 跨部门协调记录
  - 评分波动
    ↓
LLM 分析生成《阶段复盘报告》
  - 卡点热力图（哪个环节踩坑最多）
  - 效率瓶颈分析
  - 改进建议
  - 最佳实践提炼
    ↓
自动沉淀为知识资产
  - FAQ 库：常见问题 + 已验证解决方案
  - 风险预案库：类似场景的预防措施
  - 最佳实践 Wiki：高评分日报的方法论
```

**复盘报告示例**：

> **《206 样机研发 — Sprint 3 复盘报告》**
>
> - 本次迭代共发生 **12 次跨部门卡点**，其中 **8 次集中在电源选型环节**
> - 建议下季度提前引入供应商评估，将电源选型前移至设计阶段
> - 郭震在本迭代负载达 160%，建议后续分配不超过 20 点/Sprint
> - ⭐ 最佳实践：张毅的接口联调方法（先Mock再实联）使联调周期缩短 40%

**知识资产分类**：

| 资产类型 | 来源 | 用途 |
|---------|------|------|
| **FAQ 库** | 日报卡点 + 解决方案 | 新员工入职自助、AI 推荐 |
| **风险预案库** | 历史风险 + 处理措施 | 新项目规划时自动匹配 |
| **最佳实践 Wiki** | 高分日报 + 复盘亮点 | 团队学习、流程优化 |
| **供应商/技术评估** | 采购/技术复盘 | 下次选型参考 |

## 九、总结

**AI-PM 系统的核心价值**：

1. ✅ **效率革命** — 日报录入从 5 分钟 → 30 秒（结构化表单引导）
2. ✅ **质量把关** — 累加评分制质检 + 驳回修改闭环
3. ✅ **持续监督** — 跨日追踪卡点，上下游依赖预警
4. ✅ **量化管理** — 多维度评分 (0-100)，让考核有据可依
5. ✅ **全局视野** — 战情看板让管理层一目了然

> **一句话总结：用 AI 将"每日苦不堪言的填表 + 主管逐条审阅"转变为"结构化表单30秒填完 + AI 自动 PDCA 闭环管理"。**
